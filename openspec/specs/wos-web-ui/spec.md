## Requirements
### Requirement: React Web App Build
The system SHALL provide a React web application under `apps/web` that builds to static assets using Bun.

#### Scenario: Web app builds with Bun
- **WHEN** a developer runs the web build command
- **THEN** Bun SHALL build `apps/web/index.html` and its imported TypeScript, React, and CSS modules into a static output directory
- **AND** the build SHALL NOT require Vite, npm, pnpm, yarn, webpack, or a separate frontend production server

#### Scenario: Web source remains app-scoped
- **WHEN** frontend source files are added or updated
- **THEN** they SHALL live under `apps/web`
- **AND** shared non-frontend logic SHALL be imported from workspace packages instead of from `apps/cli`

### Requirement: Daemon-Served Frontend
The built web frontend SHALL be served by the wos daemon process in production from embedded executable assets when available, and from the filesystem web build output during source-checkout development.

#### Scenario: Browser requests web root
- **WHEN** a browser requests the daemon web listener root path
- **THEN** the daemon SHALL return the built `index.html`

#### Scenario: Browser requests static asset
- **WHEN** a browser requests a static asset generated by the web build
- **THEN** the daemon SHALL return the corresponding built file with an appropriate content type

#### Scenario: Browser requests frontend route
- **WHEN** a browser requests a non-file frontend route
- **THEN** the daemon SHALL return the built `index.html` as a single-page app fallback

#### Scenario: Compiled binary serves embedded frontend
- **WHEN** the daemon is running from the single-file wos executable
- **AND** no `apps/web/dist` directory is available on disk
- **THEN** browser requests to the daemon web listener SHALL be served from web UI assets embedded in the executable
- **AND** the daemon SHALL NOT return the missing web build response for normal frontend routes or generated static assets

#### Scenario: Source checkout serves filesystem frontend
- **WHEN** the daemon is running from a source checkout
- **AND** embedded web UI assets are not available
- **AND** the built `apps/web/dist` directory is available
- **THEN** browser requests to the daemon web listener SHALL be served from the filesystem build output

#### Scenario: Web build is missing
- **WHEN** the daemon web listener receives a frontend request
- **AND** neither embedded web UI assets nor built filesystem web assets are available
- **THEN** the daemon SHALL return an actionable server response explaining that the web app must be built

### Requirement: Client-Side Routing Foundation
The web frontend SHALL use `react-router` for client-side navigation within the Bun-built React app.

#### Scenario: Web app renders through router provider
- **WHEN** the web application starts in the browser
- **THEN** React SHALL render through a `react-router` router provider
- **AND** the router SHALL include a root route for `/`

#### Scenario: Frontend route can be loaded directly
- **WHEN** a browser opens a non-file frontend route served by the daemon
- **THEN** the daemon SHALL return the built `index.html`
- **AND** the client-side router SHALL render the matching route or a not-found route

#### Scenario: Main dashboard routes are available
- **WHEN** this change is implemented
- **THEN** route elements SHALL include the main project dashboard and selected worktree detail routes
- **AND** the implementation SHALL support dashboard, status, logs, and diff views for worktree inspection

### Requirement: Shadcn UI Foundation
The web frontend SHALL provide a shadcn-compatible styling and component foundation using the `quiet-workspace v3` token set: warm off-white shell, white document surface, deep ink foreground, hairline dividers, single amber accent reserved for commands, and a Geist + Geist Mono typography stack with `lucide-react` iconography.

#### Scenario: Preset theme is available globally
- **WHEN** the web application is built
- **THEN** global CSS in `apps/web/src/index.css` SHALL define the `quiet-workspace v3` token set including `--background`, `--foreground`, `--surface`, `--shell`, `--ink`, `--ink-2`, `--muted`, `--muted-foreground`, `--hair`, `--hair-2`, `--chip-bg`, `--accent`, `--accent-cmd`, `--good`, `--bad`, `--warn`, `--card`, `--border`, `--radius`
- **AND** the tokens SHALL be defined for both light and dark themes
- **AND** the global stylesheet SHALL load Geist as the sans typeface and Geist Mono as the mono typeface
- **AND** the icon convention SHALL be `lucide-react` for all chrome and content icons added or modified by this change

#### Scenario: UI primitives are app-scoped
- **WHEN** shadcn-style UI primitives are added
- **THEN** their source files SHALL live under `apps/web`
- **AND** they SHALL NOT be placed in the TUI-oriented `packages/ui` package

#### Scenario: Bun build remains authoritative
- **WHEN** a developer builds the web frontend
- **THEN** `bun run build:web` SHALL produce static assets successfully
- **AND** the build SHALL NOT require Vite, npm, pnpm, yarn, webpack, or a separate frontend production server

#### Scenario: Legacy mission-console tokens are removed
- **WHEN** the v3 token set is in place
- **THEN** the global stylesheet SHALL NOT define or rely on the prior tokens `--signal-active`, `--signal-active-soft`, `--signal-warn-soft`, `--signal-error-soft`, `--signal-info-soft`, `--grid-line`, `--grid-dot`
- **AND** no shipped component SHALL reference those token names

### Requirement: Main Project Dashboard
The web frontend SHALL render **Mission Control** as the first screen on the root route `/` (see capability `wos-mission-control`). Mission Control SHALL show a wall of live terminal-screen snapshots across all projects/worktrees; when no live terminal sessions exist it SHALL render a cross-project overview as its empty state instead of a marketing or placeholder page. The previous standalone Projects overview (`HomeRoute`) SHALL no longer be the index surface.

#### Scenario: Dashboard loads
- **WHEN** the user opens the wos web UI root route
- **THEN** the app SHALL request the unified UI API project list and the live terminal sessions snapshot
- **AND** it SHALL render a persistent left sidebar and the Mission Control area

#### Scenario: Live terminals present
- **WHEN** one or more live terminal sessions exist across registered projects
- **THEN** Mission Control SHALL render the wall of session panes per capability `wos-mission-control`

#### Scenario: No live terminals (empty-state overview)
- **WHEN** there are no live terminal sessions
- **THEN** Mission Control SHALL render every registered project as a scannable project section
- **AND** each project section SHALL show the project display name, source path, worktree count, running count, and stale or error state when present
- **AND** each project section SHALL list that project's worktrees using the same worktree label preference as the sidebar

#### Scenario: Empty-state worktree row is shown
- **WHEN** the empty-state overview renders a worktree
- **THEN** the worktree row SHALL show a branch/worktree icon, the resolved worktree label, a `root` badge for the source worktree, and a quiet deployment status glance
- **AND** the row SHALL preserve the v3 quiet-workspace visual language
- **AND** it SHALL NOT expose sidebar-only lifecycle actions, terminal creation, rename/note editing, or context-menu actions

#### Scenario: User selects a worktree from the empty-state overview
- **WHEN** the user activates a worktree row in the empty-state overview
- **THEN** the web app SHALL navigate to the existing selected worktree detail route
- **AND** it SHALL request the worktree detail from the unified UI API

#### Scenario: No projects are registered
- **WHEN** the project list is empty and there are no live terminal sessions
- **THEN** the sidebar SHALL still render an add-project action that opens the add-project modal
- **AND** the detail area SHALL present an empty state that does not require leaving the dashboard

### Requirement: Project Sidebar
The web frontend SHALL present the left rail (v3 `Rail` primitive) in two modes: a **Sessions** mode (the default) that renders a live terminal-session stream grouped by attention (see the **Rail Session Stream** requirement), and a **Worktrees** mode that renders the active project's worktree tree. The active mode SHALL be a per-browser preference selected in the Settings Appearance section (see the `wos-web-settings` **Settings Appearance Section** requirement), not a control on the rail itself; the rail SHALL render the persisted mode and SHALL re-render to the selected mode immediately when the preference changes (no reload). In **Worktrees** mode the rail SHALL scope to a single active project, SHALL provide a project switcher that lists all registered projects and lets the user switch between them, SHALL render the active project's worktree branches as an expandable tree where each branch exposes its runtime, live terminal sessions, and quick terminal creation, SHALL provide a `Projects` navigation row above `Settings`, and SHALL ground the rail with a profile footer at the bottom. The project switcher (scope) and the profile footer SHALL be shared by both modes. The rail SHALL stay within the quiet-workspace v3 visual language (`demo/side-menu-v3.html` is the canonical reference for Worktrees mode and `demo/sidebar-stream-v3.html` for Sessions mode).

#### Scenario: Rail reflects the persisted mode
- **WHEN** the rail is rendered
- **THEN** it SHALL render the body of the persisted mode, defaulting to `Sessions` when none has been persisted
- **AND** the rail SHALL NOT render a mode switch (the mode is chosen in the Settings Appearance section)
- **AND** when the Appearance mode preference changes the rail SHALL re-render to the selected mode without a reload

#### Scenario: Active project anchor is shown
- **WHEN** the unified UI API returns at least one registered project
- **THEN** the rail SHALL render a project-switcher anchor showing the active project name with a project icon and a quiet summary (the active project's worktree count and running count)
- **AND** the active project SHALL default to the project that owns the currently selected worktree, or to the first registered project when no worktree is selected

#### Scenario: User switches the active project
- **WHEN** the user opens the project switcher
- **THEN** the rail SHALL render a popover (a sheet on mobile) listing every registered project, each with a per-project health glance
- **AND** selecting a project SHALL make it the active project and re-render the rail body to that project's worktree tree
- **AND** the active project SHALL be marked as selected in the list

#### Scenario: User adds project by path
- **WHEN** the user activates `Add project` from the project switcher and submits a path
- **THEN** the web app SHALL call the unified UI API project add endpoint
- **AND** it SHALL refresh the project list after the add succeeds

#### Scenario: Add project fails
- **WHEN** the unified UI API rejects a submitted project path
- **THEN** the web app SHALL display the validation message near the add-project field
- **AND** it SHALL keep the user's submitted path available for correction

#### Scenario: Projects navigation row is shown
- **WHEN** the rail is rendered
- **THEN** it SHALL render a `Projects` navigation row that links to the home route `/`
- **AND** the `Projects` row SHALL appear above `Settings`
- **AND** the row SHALL use the rail-active background when the current route is `/`

#### Scenario: Global actions remain available
- **WHEN** the rail is rendered
- **THEN** `Projects`, `New worktree` (keyboard hint `⌘N`), `Automations`, and `Customize` SHALL remain reachable from the rail regardless of which project or worktree is active
- **AND** `New worktree` SHALL target the active project

#### Scenario: Active project worktree tree is rendered
- **WHEN** the rail is in Worktrees mode and the active project has discovered worktree branches
- **THEN** the rail SHALL render each branch as a worktree row with the branch name in Geist Mono and a leading worktree (branch) icon
- **AND** the currently selected worktree SHALL use the rail-active background
- **AND** worktree rows SHALL NOT render a deployment status dot or a deployment status word

#### Scenario: Branch expands to its runtime, sessions, and terminal creation
- **WHEN** the rail is in Worktrees mode and a worktree branch has a runtime or at least one live terminal session
- **THEN** the branch row SHALL be expandable, and the selected worktree SHALL be expanded by default
- **AND** an expanded branch SHALL render, in order: a single `Runtime` line, one row per live terminal/agent session, and a `New terminal` row
- **AND** the rail SHALL NOT render individual service rows, ports, tunnels, or logs (those remain in the Runtime panel)

#### Scenario: Runtime line carries deployment status
- **WHEN** a branch is expanded
- **THEN** its `Runtime` line SHALL express the deployment status as a leading dot plus a status word with a quiet service summary
- **AND** activating the `Runtime` line SHALL hand off to the Runtime panel for that worktree

#### Scenario: Terminal count is hidden while a branch is expanded
- **WHEN** a worktree branch is collapsed and has at least one live terminal session
- **THEN** the branch row SHALL show a terminal count indicator
- **WHEN** the branch is expanded
- **THEN** the branch row SHALL NOT show the terminal count indicator

#### Scenario: User creates a terminal quickly
- **WHEN** the rail is in Worktrees mode
- **THEN** each worktree branch SHALL expose a quick `New terminal` action (as a row in its expansion and as an action on branch-row hover on desktop)
- **AND** activating it SHALL create a terminal session for that worktree

#### Scenario: User attaches to or kills a session
- **WHEN** a live terminal/agent session row is rendered under a branch
- **THEN** activating the row SHALL attach to that session within its worktree
- **AND** a kill action SHALL terminate that session

#### Scenario: Profile footer is rendered
- **WHEN** the rail is rendered
- **THEN** the rail SHALL render a footer area with an avatar (initials of the current user, or a fallback marker for the local mode), a name line, and a plan/mode line
- **AND** the footer SHALL be visible without scrolling the rail when there is at least one project visible

#### Scenario: User selects a worktree
- **WHEN** the user clicks a worktree in the sidebar
- **THEN** the web app SHALL navigate to or otherwise select that worktree
- **AND** it SHALL request the worktree detail from the unified UI API

#### Scenario: Rail content is reachable on mobile via the bottom-sheet navigator
- **WHEN** the viewport is a touch/mobile size
- **THEN** the rail's project switcher and active-mode body SHALL be presented through the mobile bottom-sheet navigator (see the **Mobile Bottom Navigation** requirement), not a left slide-in drawer, with touch row heights of at least 44px
- **AND** the primary row action SHALL be visible without hover
- **AND** the project switcher SHALL open as a sheet

### Requirement: Worktree Selection
The web frontend SHALL show detail for a selected worktree when the user selects it from the sidebar, and SHALL keep the selected branch's runtime status current in the rail without a full page reload.

#### Scenario: User selects worktree
- **WHEN** the user clicks a worktree in the sidebar
- **THEN** the web app SHALL navigate to or otherwise select that worktree
- **AND** it SHALL request the worktree detail from the unified UI API

#### Scenario: Worktree status changes
- **WHEN** a worktree's deployment status changes because an operation starts, succeeds, fails, or status refresh completes
- **THEN** the branch's `Runtime` line in the rail SHALL update its status dot and word without requiring a full page reload
- **AND** the rail SHALL update the active project's running count to stay consistent with the worktree tree

### Requirement: Worktree Launch Action
The web frontend SHALL present a not-started document surface for selected worktrees that have not been initialized with `wos up`, and SHALL transition the selected worktree into a structured in-progress deployment view as soon as launch is accepted.

#### Scenario: Selected worktree is not started
- **WHEN** the selected worktree detail reports deployment status `not_started`
- **AND** there is no active running `up` operation for that worktree session
- **THEN** the main detail area SHALL render the not-started document with: an `H1` containing the branch name, a prose description of what `Start worktree` will do, an option list (for example `Run migrations`, `Reset database`, `Open tunnel for web`) and an action row containing a solid `Start worktree` button alongside secondary actions
- **AND** activating `Start worktree` SHALL submit an `up` operation through the unified UI API

#### Scenario: Launch operation starts
- **WHEN** the `Start worktree` action successfully submits an `up` operation
- **THEN** the selected worktree view SHALL immediately show the operation as in progress using `pending` status
- **AND** it SHALL leave the not-started document surface
- **AND** it SHALL render the deploying document surface (command-pill, todo-banner with a spinning loader, numbered step list, live log tail) rather than a default full log panel
- **AND** it SHALL keep init log output visible inline in the live log tail while first-run setup or init-script work is active
- **AND** it SHALL stream or poll operation updates through the unified UI API

#### Scenario: Launch operation is already active on detail load
- **WHEN** the selected worktree detail reports deployment status `pending`
- **AND** the active operation kind is `up`
- **THEN** the web app SHALL render the deploying document surface instead of the not-started document surface
- **AND** the deploying surface SHALL show the numbered step list and the contextual init log tail without requiring a user-selected log channel

#### Scenario: Launch submission fails
- **WHEN** the user submits the `Start worktree` action
- **AND** the unified UI API rejects the request
- **THEN** the web app SHALL display the API error message in the not-started document surface
- **AND** it SHALL preserve the previous not-started snapshot state

### Requirement: First Launch Init Progress
The web frontend SHALL show first-run setup and init-script progress for a selected worktree whose `up` operation is active before service discovery has completed, using the deploying document surface with a todo-banner, a numbered step list, and an inline live log tail.

#### Scenario: First-run setup starts
- **WHEN** the web UI receives a `deployment.step` event for the selected session with step `first-run-setup` and state `running`
- **THEN** the deploying document surface SHALL render the corresponding step entry in the numbered step list with a spinning loader marker and the running label
- **AND** the live log tail under the step list SHALL include init log output while that step is running

#### Scenario: Init script starts
- **WHEN** the web UI receives a `deployment.step` event for the selected session with step `init-script` and state `running`
- **THEN** the deploying document surface SHALL render the corresponding step entry as running with a spinning loader marker
- **AND** init log chunks from the unified UI API SHALL appear in the live log tail while initialization is running

#### Scenario: Services are not discovered yet
- **WHEN** a selected worktree has an active running `up` operation
- **AND** the worktree detail response contains no managed services yet
- **THEN** the deploying document surface SHALL render the todo-banner and step list communicating an in-progress deployment state
- **AND** it SHALL NOT present an empty service list as a final no-services state

#### Scenario: Init step succeeds
- **WHEN** the web UI receives a `deployment.step` event for the selected session with step `first-run-setup` or `init-script` and state `done`
- **THEN** the deploying document surface SHALL mark that step as complete in the step list
- **AND** when the overall `up` operation completes, the completed init log output SHALL be reachable through a collapsed environment-section affordance so it does not dominate the subsequent overview layout

#### Scenario: Init step fails
- **WHEN** the web UI receives a `deployment.step` event for the selected session with step `first-run-setup` or `init-script` and state `failed`
- **THEN** the selected worktree view SHALL switch to the failed document surface and mark the failed step with the danger color in the step list
- **AND** it SHALL expose the init log channel through the explicit `Open log` action in the failed surface without requiring the user to switch through service selection

### Requirement: Worktree Status View
The web frontend SHALL render selected-worktree status information as a document surface: a header with branch name, a command-pill describing the active or last command, a todo-style status banner, and the same data points (services, addresses, healthchecks, tunnels) that the previous status view rendered.

#### Scenario: Initialized worktree selected
- **WHEN** the selected worktree detail includes services
- **THEN** the web app SHALL render service names, service states, published addresses, app-port healthcheck state, and tunnel information when present
- **AND** service state and healthcheck state SHALL be expressed through a leading status dot color and inline text (for example `running · healthy`) rather than through bordered chip badges

#### Scenario: Status is stale or unknown
- **WHEN** the unified UI API reports stale or unknown status for a selected worktree
- **THEN** the web app SHALL render that state visibly in the document header without hiding the last available worktree identity information

#### Scenario: Service count is available
- **WHEN** the selected worktree detail includes a service summary
- **THEN** the web app SHALL render the running service count next to deployment status in the form `<running> of <total>`
- **AND** the running count SHALL appear in the document todo-banner (for example `4 of 4 services running`) and SHALL stay consistent with the count surfaced in the worktree top bar and in the rail

### Requirement: Worktree Detail Full-Width Tabs
The web frontend SHALL render the worktree detail view as a single content surface with `Overview`, `Runtime`, `Review`, `Files`, and `Terminal` tabs, where exactly one tab is the active destination. Reaching any destination SHALL NOT require opening, closing, resizing, or docking an internal per-tab right-side panel within the content area. The worktree detail view is host-agnostic: it renders the same tabs and surfaces in the full-screen `/worktree` page host and in the docked panel host (see Worktree Detail Panel Host). Any host-level close, resize, or expand control belongs to the panel host chrome — the panel's resize edge and the compact header's inline expand/close controls — never an internal per-tab right-side panel; the full-screen page host renders none of these controls.

#### Scenario: Worktree detail renders tab navigation
- **WHEN** the worktree detail view is rendered in either host
- **THEN** the web app SHALL render worktree tab navigation with `Overview`, `Runtime`, `Review`, `Files`, and `Terminal`
- **AND** exactly one worktree tab SHALL be the active content destination
- **AND** the active tab SHALL render in the main worktree content area

#### Scenario: Worktree detail view is host-agnostic
- **WHEN** the same worktree is rendered in the full-screen page host and then in the docked panel host
- **THEN** both hosts SHALL render the same tab set, active-tab selection model, and tab surfaces
- **AND** the hosts SHALL differ only in their surrounding chrome (page host is full screen; panel host adds close, resize, and expand controls)

#### Scenario: Overview is selected
- **WHEN** the active worktree tab is `Overview`
- **THEN** the web app SHALL render the selected worktree overview dossier as the main content
- **AND** it SHALL show the overview identity, intent/note, branch facts, change summary, sessions summary, runtime summary, and continue actions

#### Scenario: Runtime is selected
- **WHEN** the active worktree tab is `Runtime`
- **THEN** the web app SHALL render the full deployment lifecycle surface as the main content
- **AND** the surface SHALL include launch setup, deployment progress, failure recovery, running services, tunnels, lifecycle controls, and channel-scoped logs according to the selected worktree state

#### Scenario: Review is selected
- **WHEN** the active worktree tab is `Review`
- **THEN** the web app SHALL render staged and unstaged review diffs as the main content
- **AND** it SHALL preserve review mode switching, file pagination, empty states, syntax highlighting, and review dirty totals

#### Scenario: Files is selected
- **WHEN** the active worktree tab is `Files`
- **THEN** the web app SHALL render the file explorer as the main content for the selected worktree

#### Scenario: Terminal is selected
- **WHEN** the active worktree tab is `Terminal`
- **THEN** the web app SHALL render the worktree terminal sessions surface as the main content
- **AND** it SHALL preserve terminal session creation, attach, rename, focus, and empty-state behavior

#### Scenario: User switches worktree tabs
- **WHEN** the user activates a different worktree tab
- **THEN** the web app SHALL replace the main content with the selected tab body
- **AND** it SHALL preserve selected worktree context
- **AND** it SHALL NOT create a user-defined tab record
- **AND** it SHALL NOT open or close an internal per-tab right-side panel

#### Scenario: Page host renders no host-level panel controls
- **WHEN** the worktree detail view is rendered in the full-screen `/worktree` page host on a desktop viewport
- **THEN** the web app SHALL NOT render a host-level close, resize, or expand control (the page host already fills the screen)
- **AND** the worktree tab surface SHALL NOT render an internal per-tab right-side panel, nor require a docked panel width to access Runtime, Review, Files, or Terminal

#### Scenario: Worktree chrome is visually distinct
- **WHEN** the worktree detail view is rendered on a desktop viewport
- **THEN** the worktree tab navigation SHALL be visually distinct from the identity header above it
- **AND** the active worktree tab SHALL be visually distinct from the inactive tabs

#### Scenario: Persisted right-panel state is migrated
- **WHEN** the browser has persisted right-panel visibility with `open: true` and a supported tab
- **THEN** the worktree detail view SHALL restore the equivalent active worktree tab
- **AND** a legacy persisted `logs` tab SHALL restore as `Runtime`
- **WHEN** the browser has persisted right-panel visibility with `open: false`
- **THEN** the worktree detail view SHALL restore `Overview` as the active worktree tab

#### Scenario: Legacy panel query param selects a tab
- **WHEN** the full-screen `/worktree` route is opened with `panel=runtime`, `panel=review`, `panel=files`, or `panel=terminal`
- **THEN** the web app SHALL select the corresponding full-width worktree tab
- **AND** it SHALL remove the one-shot `panel` param after applying it

#### Scenario: Legacy logs query param selects Runtime
- **WHEN** the full-screen `/worktree` route is opened with legacy `panel=logs`
- **THEN** the web app SHALL select the `Runtime` tab
- **AND** it SHALL make the init log channel available without requiring a right-side log panel
- **AND** it SHALL remove the one-shot `panel` param after applying it

#### Scenario: Terminal query param focuses a session
- **WHEN** the full-screen `/worktree` route is opened with `terminal=<id>`
- **THEN** the web app SHALL select the `Terminal` tab
- **AND** it SHALL focus the requested terminal session as a one-shot request
- **AND** it SHALL remove the one-shot `terminal` param after applying it

### Requirement: Worktree Logs And Diffs
The web frontend SHALL provide a channel-scoped log view for the selected worktree inside the full-width Runtime tab and SHALL expose worktree changes through the full-width Review tab as a changed-files explorer with a single-file diff and a commit composer, instead of a right-side Review sidebar, an old Diff tab, or staged/unstaged mode tabs. Review controls SHALL show change totals in `+N` (green) and `-N` (red) text when dirty. The Review tab's changed-files explorer and commit composer SHALL remain mounted whether or not the worktree has changes, so the user always has access to branch posture and sync actions.

#### Scenario: User opens init logs
- **WHEN** the user selects the init log entry in the selected worktree view
- **THEN** the web app SHALL select the `Runtime` tab
- **AND** it SHALL mark the init entry active
- **AND** it SHALL show buffered and live log chunks only from the `init` channel in the Runtime tab log region

#### Scenario: User opens service logs
- **WHEN** the user clicks the explicit `Open logs` affordance on a service row in the selected worktree service list
- **THEN** the web app SHALL select the `Runtime` tab
- **AND** it SHALL mark that service active
- **AND** it SHALL show buffered and live log chunks only from that service's `service:<name>` channel in the Runtime tab log region

#### Scenario: Active service remains visible
- **WHEN** the selected worktree service list renders an active service
- **THEN** the active service row SHALL have a distinct selected visual state separate from hover and disabled action states

#### Scenario: Selected channel has no output
- **WHEN** the selected init or service log channel has no buffered or live output
- **THEN** the web app SHALL show an empty log state for that selected channel
- **AND** it SHALL NOT display log chunks from other init or service channels

#### Scenario: Review button shows change totals
- **WHEN** the selected worktree detail view is rendered
- **THEN** the worktree tab navigation or worktree action area SHALL include a Review control
- **AND** the Review control SHALL show aggregate additions and deletions across staged and unstaged changes in the form `Review +N -N` where `+N` uses the `--good` token colour and `-N` uses the `--bad` token colour

#### Scenario: User opens review
- **WHEN** the user activates a Review control
- **THEN** the web app SHALL select the full-width `Review` tab for the selected worktree
- **AND** Runtime, Files, Terminal, and Overview SHALL remain available through worktree tab navigation

#### Scenario: Review renders a changed-files explorer with a single-file diff
- **WHEN** the Review tab is active and structured diff data is available
- **THEN** the Review tab SHALL render a single changed-files explorer list combining staged and unstaged changes
- **AND** selecting a file SHALL render that file's diff in a detail pane with its path, status, additions, deletions, hunks, line gutters, and added / deleted / context line styling
- **AND** textual diff line content SHALL use syntax highlighting when a CodeMirror language can be detected from the changed file path
- **AND** textual diff line content SHALL remain readable as plain text when no matching language is available
- **AND** non-text and binary file rows SHALL render as file-level review states instead of syntax-highlighted code

#### Scenario: Review offers a single-file and an all-files view
- **WHEN** the Review tab is active
- **THEN** the Review tab SHALL provide a toggle between the single-file explorer view and an all-files unified view
- **AND** the all-files view SHALL render every changed file as a stacked section with sticky file headers
- **AND** the Review tab SHALL provide an inline / split toggle for the rendered diff

#### Scenario: Review explorer scrolls all changed files
- **WHEN** the active changed-files explorer contains more files than fit in the viewport
- **THEN** the Review tab SHALL render the full changed-files list in a scrollable explorer without paginating it into pages

#### Scenario: Review collapses unchanged context
- **WHEN** a rendered diff contains a run of unchanged context lines longer than the visible context threshold
- **THEN** the web app SHALL replace the hidden run with a collapsed row labeled with the number of hidden unchanged lines, such as `13 unmodified lines`
- **AND** the collapsed row SHALL be visually distinct from added, deleted, and normal context lines

#### Scenario: Worktree has no changes
- **WHEN** the selected worktree has no staged or unstaged changes

#### Scenario: Panel resize handle is accessible
- **WHEN** the docked panel resize handle is rendered
- **THEN** it SHALL expose separator semantics with vertical orientation and current/minimum/maximum width values
- **AND** it SHALL have an accessible label that identifies it as the worktree panel resize control

### Requirement: Container-Aware Worktree Chrome
The worktree detail chrome (the tab strip, identity header, and git posture line) SHALL adapt to the width of its host container rather than to the browser viewport width, so a narrow panel host renders a compact layout instead of the full-width page layout.

#### Scenario: Narrow panel renders compact chrome
- **WHEN** the worktree detail view is hosted in a docked panel that is narrower than the full-width page layout while the viewport is wide
- **THEN** the worktree chrome SHALL render a compact layout that fits the panel width without horizontal overflow
- **AND** it SHALL keep all five tab destinations (`Overview`, `Runtime`, `Review`, `Files`, `Terminal`) reachable

#### Scenario: Full-width host renders full chrome
- **WHEN** the worktree detail view is hosted in the full-screen `/worktree` page host
- **THEN** the worktree chrome SHALL render its full-width layout including the identity and git posture segments that the page host has space for

### Requirement: Review Branch Sync Posture
The Review tab SHALL render an always-visible branch sync posture row beneath its toolbar that shows the branch's upstream tracking state and exposes `Fetch` and `Push` actions. The row is the single source of truth for ahead/behind posture in the Review tab. It SHALL stay within the quiet-workspace v3 visual language: state expressed as a leading dot plus inline word, ahead/behind counts and the `↑` / `↓` glyphs rendered as text in the `--ink-2` / `--good` / `--bad` tokens, a `RefreshCw` spinner for in-flight operations, and no bordered status chips and no amber (amber is reserved for slash-commands). To avoid duplicating posture, the commit composer's branch line SHALL NOT repeat the `↑N ahead` count while this row is present, except for the detached-HEAD create-branch flow which is unchanged.

#### Scenario: Posture row reflects ahead and behind counts
- **WHEN** the Review tab is active for a branch that tracks an upstream
- **THEN** the posture row SHALL show the branch (or `origin/<branch>`) and its `↑ahead` and `↓behind` counts using text tokens
- **AND** when the branch is level with its upstream it SHALL show an up-to-date state with a leading dot and word rather than a count

#### Scenario: Push is gated on having unpushed commits
- **WHEN** the branch has no commits ahead of its upstream
- **THEN** the `Push` action SHALL be disabled
- **WHEN** the branch has at least one commit ahead of its upstream
- **THEN** the `Push` action SHALL be enabled

#### Scenario: Fetch refreshes posture
- **WHEN** the user activates `Fetch`
- **THEN** the web app SHALL call the fetch endpoint and show an in-flight spinner state on the row
- **AND** on success it SHALL update the displayed ahead/behind counts from the response and record a client-side `fetched <relative time> ago` freshness label
- **AND** it SHALL trigger a worktree-detail reload so other surfaces (such as the Overview ledger) reconcile

#### Scenario: Push updates posture
- **WHEN** the user activates `Push` for a branch with unpushed commits
- **THEN** the web app SHALL call the push endpoint and show an in-flight spinner state on the row
- **AND** on success it SHALL update the displayed ahead/behind counts from the response and trigger a worktree-detail reload

#### Scenario: Push is rejected as non-fast-forward
- **WHEN** the push endpoint returns a non-fast-forward rejection
- **THEN** the web app SHALL surface a toast prompting the user to fetch first
- **AND** it SHALL leave the posture row in its prior state

#### Scenario: Branch has no upstream
- **WHEN** the Review tab is active for a branch with no upstream or a detached HEAD
- **THEN** the posture row SHALL show a `no upstream` state and SHALL omit ahead/behind counts
- **AND** the `Fetch` action SHALL remain available

#### Scenario: One sync operation at a time
- **WHEN** a fetch or push operation is in flight
- **THEN** both the `Fetch` and `Push` actions SHALL be disabled until it resolves

#### Scenario: Compact host collapses the posture row
- **WHEN** the Review tab is hosted in a container narrower than the full-width page layout
- **THEN** the posture row SHALL render as a single compact line without horizontal overflow

### Requirement: Terminal Backend Stability Warning
When the active terminal backend is `default`, the web frontend SHALL surface a stability warning at terminal-session creation, presented as a local v3 warning accent near the affected session and never as global page chrome. The warning copy SHALL be `Running outside tmux/psmux — terminal sessions may be unstable.`. When the active backend is `tmux` the web frontend SHALL NOT show the warning.

#### Scenario: Warning shown when creating a terminal on the default backend
- **WHEN** the user creates a terminal session and the active terminal backend is `default`
- **THEN** the web UI SHALL display `Running outside tmux/psmux — terminal sessions may be unstable.` as a local accent associated with the created session

#### Scenario: Warning uses a local accent, not global chrome
- **WHEN** the stability warning is shown
- **THEN** it SHALL be rendered as a local v3 warning accent (for example a dot plus word or a soft inline note) near the session
- **AND** it SHALL NOT be rendered as a global banner or persistent page chrome

#### Scenario: No warning on the tmux backend
- **WHEN** the user creates a terminal session and the active terminal backend is `tmux`
- **THEN** the web UI SHALL NOT display the outside-tmux/psmux stability warning

### Requirement: Web Terminal Text Selection And Copy
The web frontend SHALL let users select terminal text and copy it to the system clipboard even while a foreground application (such as tmux with mouse reporting, or a full-screen TUI) holds mouse tracking, and SHALL honor application-driven clipboard writes through the terminal emulator.

#### Scenario: Force a native selection while an app tracks the mouse
- **WHEN** a foreground application has mouse tracking active in the terminal viewport
- **AND** the user holds the platform selection-override modifier (Option on macOS, Shift on other platforms) and drags across terminal text
- **THEN** the viewport SHALL create a native text selection of the dragged region
- **AND** it SHALL NOT forward that drag to the foreground application

#### Scenario: Copy the selected text
- **WHEN** the terminal viewport has a native text selection
- **AND** the user issues the platform copy shortcut
- **THEN** the web frontend SHALL place the selected text on the system clipboard

#### Scenario: Drag without the modifier is unchanged
- **WHEN** a foreground application has mouse tracking active
- **AND** the user drags without the selection-override modifier
- **THEN** the viewport SHALL forward the drag to the application as before and SHALL NOT create a native selection

#### Scenario: Application sets the clipboard via OSC 52
- **WHEN** a terminal application emits an OSC 52 clipboard-write sequence
- **THEN** the web frontend SHALL write the provided text to the system clipboard through the emulator's clipboard integration

### Requirement: Web Terminal On-Screen Keyboard Visibility
The web frontend SHALL keep the terminal's foreground input line visible above the on-screen keyboard when a controller types directly into a touch-oriented terminal, by tracking the visual viewport and fitting the terminal to the keyboard-reduced area. This behavior is scoped to direct typing into the terminal (the Write Composer closed) and SHALL NOT alter the Write Composer or viewer behavior.

#### Scenario: Keyboard appears while a controller types
- **WHEN** the on-screen keyboard appears while the controller is focused on a touch-oriented terminal and the Write Composer is closed
- **THEN** the web frontend SHALL constrain the terminal viewport to the keyboard-reduced visible area tracked from the visual viewport
- **AND** it SHALL refit the terminal so the foreground program's bottom rows, including its input line, remain visible above the keyboard

#### Scenario: Keyboard is dismissed
- **WHEN** the on-screen keyboard is dismissed
- **THEN** the web frontend SHALL restore the terminal viewport to the full available area and refit

#### Scenario: Keyboard fit is a single debounced resize
- **WHEN** the terminal viewport is fitted for the on-screen keyboard
- **THEN** the resulting PTY resize SHALL be debounced to a single settle rather than continuous resize churn

#### Scenario: Viewer does not resize for the keyboard
- **WHEN** a non-controller attachment views a touch-oriented terminal
- **THEN** visual-viewport changes SHALL NOT send PTY resize commands
- **AND** the viewer SHALL continue rendering at the session's current dimensions

#### Scenario: Browser resizes the layout viewport for the keyboard
- **WHEN** the web app declares content-resize interactive-widget viewport behavior
- **AND** it runs in a browser that resizes the layout viewport when the on-screen keyboard appears
- **THEN** the existing resize-and-fit path SHALL keep the terminal fitted to the reduced area without additional handling

### Requirement: Rail Session Stream
In Sessions mode the rail SHALL render every live terminal/agent session in the current scope as a flat stream grouped by attention into ordered groups — **Needs you**, **Unread**, **Working**, **Idle** — each session appearing in exactly one group. A session SHALL be classified by precedence: agent activity `awaiting-input` → Needs you (even when also unread); else unread output present → Unread; else agent activity `working` → Working; else Idle. The scope SHALL filter the stream: `Active now` SHALL include the live sessions of every project, and a selected project SHALL include only that project's sessions. Each non-empty group SHALL render a header with the group name and its session count. The stream SHALL derive entirely from the already-aggregated live-session snapshot and agent activity/telemetry; it introduces no new daemon or API data and shows only live (`running`/`creating`) sessions.

#### Scenario: Sessions are grouped by attention
- **WHEN** the rail is in Sessions mode with live sessions in scope
- **THEN** the rail SHALL render the non-empty groups in the order Needs you, Unread, Working, Idle
- **AND** each group SHALL show its name and the count of sessions it contains

#### Scenario: A blocked agent outranks its unread output
- **WHEN** a session is both awaiting input and has unread output
- **THEN** it SHALL appear in the Needs you group and SHALL NOT also appear in Unread

#### Scenario: Each session row shows project identity, agent, and worktree
- **WHEN** a session row is rendered in the stream
- **THEN** it SHALL show a colored project identity tile (a monogram of the project, deterministically colored), the agent glyph (brand-tinted for a known agent, else a shell glyph), the session title, and on a second line the worktree name (and the active command for a plain shell)
- **AND** the row SHALL NOT render a per-row deployment/activity status dot (the group conveys the state)
- **AND** the project name SHALL NOT be repeated as text on the row

#### Scenario: A working session animates its project tile
- **WHEN** a session's agent activity is `working`
- **THEN** its project tile SHALL show a running animation around the tile border in the project color, and the tile SHALL remain a rounded square
- **WHEN** the user prefers reduced motion
- **THEN** the tile SHALL render without animation

#### Scenario: Unread sessions use the canonical unread modifier
- **WHEN** a session has unread output and is in the Unread group
- **THEN** its name SHALL render bold with a small filled blue trailing dot (the canonical `--unread` modifier)

#### Scenario: Agent sessions show telemetry; a blocked agent shows its question
- **WHEN** a session has meaningful agent telemetry
- **THEN** the row SHALL render the shared telemetry cluster (a context meter plus context tokens in use and total token spend, reddening past the warn threshold) with the same hover popover used by the worktree-tree session row
- **WHEN** a session is awaiting input with a permission question
- **THEN** the row SHALL render the question summary on an amber line

#### Scenario: Filter narrows the stream to one group
- **WHEN** the user selects a filter (All, Needs you, Unread, Working, or Idle)
- **THEN** `All` SHALL show every non-empty group and any other selection SHALL show only that group
- **AND** each filter SHALL carry the count of sessions it would show

#### Scenario: User attaches to or kills a stream session
- **WHEN** a session row in the stream is activated
- **THEN** the web app SHALL attach to that session
- **AND** a kill action on the row SHALL terminate that session

#### Scenario: User starts a new session from the stream
- **WHEN** the user activates the New session launcher
- **THEN** the rail SHALL let the user pick a target worktree (grouped by project) and create a terminal session in it, then attach
- **AND** `Add project` SHALL be reachable from the scope/project dropdown rather than as a standalone rail button

#### Scenario: Empty scope is handled
- **WHEN** Sessions mode has no live sessions in the current scope
- **THEN** the rail SHALL render a quiet empty state rather than blank space
